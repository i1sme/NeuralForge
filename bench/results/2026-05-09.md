## Findings

- **`classifier` and `self_attention`: arm64 wins** by 1.6× and 1.8× respectively. Expected — arm64 codegen uses single-rounding FMA (`fmadd`) per inner-loop iteration; x86_64 uses two-instruction `mulss + addss` (intentional ISA divergence to keep SSE2-only as the floor). On matmul-mass and dispatch-overhead-bound workloads, FMA's per-instruction throughput dominates.
- **`large_classifier_k`: x86_64 wins** by 1.4×. **Unexpected** — same FMA-vs-mulss-addss ISA divergence should favour arm64 here too, since K=8192 is the inner reduction. The reversal suggests one or more of: (a) `large_classifier_k`'s shape `[2, 8192] @ [8192, 10]` makes the K-loop hit a memory-pattern that M1's prefetcher handles less well than ubuntu-latest's Xeon, (b) the small output dim (10) means loop-iteration overhead dominates per-output, where x86_64's simpler 2-instruction cadence happens to schedule better, (c) M1 runner co-tenancy noise (p95 = 360.9 µs vs median 219.1 µs is a 1.65× tail; x86_64's tail is 1.15×, much tighter).
- **arm64 noise is higher.** p95/median ratios — arm64: 1.25× / 1.65× / 1.05×; x86_64: 1.01× / 1.15× / 1.31×. arm64's `large_classifier_k` is the loudest tail (1.65×), which weakens the arm64-slower conclusion: a less noisy run might land closer to parity. Future SIMD work should re-bench to see if vectorised inner loops change the picture.

*First M11 combined report. Numbers are noisy on shared GitHub-hosted runners; treat absolute values as informational only. Cross-run comparisons are unreliable; comparisons within this single run are reliable. The arm64 / x86_64 ratio for `large_classifier_k` should be re-validated in the next bench cycle (after Axis 1 SIMD work, or after a noise-reducing CI configuration is found).*
