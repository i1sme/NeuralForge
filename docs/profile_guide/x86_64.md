# `x86_64` Profile — Linux ELF Scalar SSE2 Codegen

> **Status:** M13 complete (add op + N=4 matmul fix). Lowers `linear` (with or without
> `bias=true`), `relu`, `dropout` (no-op pass-through at inference),
> `softmax` (numerically stable 3-pass via libm `expf@PLT`,
> generalised to rank ≥ 2 in M10), `linear → softmax` fused via
> `PostOp::SoftmaxRow`, and the two M10 attention-pattern ops `matmul`
> (multi-dim, optional `transpose_b`) and `mul_scalar` (scalar
> pre-load + flat loop) to scalar x86_64 AT&T assembly targeting Linux
> ELF. The compiler runs the default UIR-pass pipeline
> (`EliminateDropout` + `FuseLinearRelu` + `FuseLinearSoftmax`) before
> lowering.
> **Authoritative source:** `profiles/x86_64/src/` and the M9 / M10 / M12
> specs under `docs/superpowers/specs/`.

The `x86_64` profile is the second concrete codegen profile in NeuralForge,
landing in M9. It takes a `compiler::Uir` and emits x86_64 scalar assembly
(AT&T syntax, Linux ELF / SysV AMD64 ABI) callable as a C function. It
implements the shared `Profile` trait from `profile-api/`, coexisting with
`profiles/arm64/` under a single `nflc compile --profile <name>` dispatch.
Op-level parity with arm64 is complete; SIMD/AVX extensions and macOS
Mach-O support are explicitly out of scope for this profile — see §8.

As of M9, arm64 coexists with the x86_64 Linux ELF profile. Both implement
the shared `Profile` trait from `profile-api/`; the symbol-prefix abstraction
(`sym_prefix() -> "_"` on Mach-O, `""` on ELF) plus per-profile asm emission
is the contract. Cross-profile architectural differences (notably: callee-saved
FP register sets) are documented in this file; see also [`arm64.md`](arm64.md).

---

## 1. Overview

The x86_64 profile targets **Linux ELF, scalar SSE2**. It emits AT&T syntax
assembly suitable for consumption by `gas` / `cc` on any x86_64 Linux host.
No SIMD intrinsics are used; all computation is scalar (single `%xmm`
registers in their lower 32-bit `float` lane).

**Positioning vs arm64.** The arm64 profile targets Apple Silicon / AArch64
POSIX (Mach-O). The x86_64 profile targets Linux ELF (SysV AMD64). The two
profiles share:

- The `Asm`, `FnSig`, `ParamSlot`, `ParamKind`, `LowerError` types from
  `profile-api/`.
- The `Profile` trait interface (`lower` + `sym_prefix`).
- The UIR-pass pipeline (`EliminateDropout`, `FuseLinearRelu`,
  `FuseLinearSoftmax`) run by `nflc` before calling `lower`.
- The same set of supported ops (linear ± bias, relu, dropout, softmax,
  ReluFused, SoftmaxRow).

They diverge in: ABI (AAPCS64 vs SysV AMD64), register conventions
(callee-saved FP on arm64, none on x86_64 — see §3), libm call form
(`bl _expf` on Mach-O, `call expf@PLT` on ELF — see §6), and stack
alignment accounting (see §7).

## 2. ABI (SysV AMD64)

For each `UirModel` in the input UIR, the profile emits one `extern "C"` function:

```c
void nfl_forward_<ModelName>(
    const float* input,
    const float* params,    // packed: weights + biases of all Linear nodes
    float*       output
);
```

**SysV AMD64 argument passing.** The first six integer/pointer arguments are
passed in registers, in order: `%rdi`, `%rsi`, `%rdx`, `%rcx`, `%r8`, `%r9`.
Our 3-pointer signature maps as:

| Parameter | Register |
|-----------|----------|
| `input`   | `%rdi`   |
| `params`  | `%rsi`   |
| `output`  | `%rdx`   |

The symbol name in the asm is `nfl_forward_<ModelName>` (no underscore prefix).
On Linux ELF, C / FFI callers call the symbol directly; `dlsym` returns the
function pointer without any loader-side renaming. Compare with arm64 Mach-O,
where the symbol is `_nfl_forward_<ModelName>` and C callers pass the
underscore-less name to `dlsym`. The `Profile::sym_prefix()` method returns
`""` for this profile and `"_"` for arm64.

**Callee-saved registers.** The callee must preserve the following across
the call:

- Integer: `%rbx`, `%rbp`, `%r12`, `%r13`, `%r14`, `%r15`.
- FP: **none** — see §3 for the critical divergence from arm64.

**Stack alignment at call boundary.** At every `call` instruction, `%rsp`
must be 16-byte aligned. The SysV ABI specifies this as a requirement on the
_caller_ at the point of `call`. See §7 for how `compute_frame_size` maintains
this invariant throughout the prologue.

**Return value.** The function returns `void`; results are written via the
output pointer. Callers own all three buffers.

## 3. Register Conventions

### Integer registers

| Register | Role in this profile                             | Caller/callee saved |
|----------|--------------------------------------------------|---------------------|
| `%rdi`   | `input` pointer (arg 0)                          | caller              |
| `%rsi`   | `params` pointer (arg 1)                         | caller              |
| `%rdx`   | `output` pointer (arg 2)                         | caller              |
| `%rax`   | scratch / call result                            | caller              |
| `%rcx`   | loop counter / scratch                           | caller              |
| `%r8`    | scratch (inner loop)                             | caller              |
| `%r9`    | scratch (inner loop)                             | caller              |
| `%r10`   | dimension immediate (B, N, K)                    | caller              |
| `%r11`   | scratch (address computation)                    | caller              |
| `%rbx`   | outer loop index `i` across `call expf@PLT`      | **callee** (saved)  |
| `%rbp`   | always pushed (frame pointer / alignment)        | **callee** (saved)  |
| `%r12`   | src pointer (surviving `call expf@PLT`)          | **callee** (saved)  |
| `%r13`   | dst pointer (surviving `call expf@PLT`)          | **callee** (saved)  |
| `%r14`   | weight pointer (surviving `call expf@PLT`)       | **callee** (saved)  |
| `%r15`   | inner loop index / row offset                   | **callee** (saved)  |

### Float registers

| Register  | Role                                             | Caller/callee saved |
|-----------|--------------------------------------------------|---------------------|
| `%xmm0`   | arg to `expf` / scalar accumulator              | caller              |
| `%xmm1`–`%xmm7` | temporaries                               | caller              |
| `%xmm8`–`%xmm15` | temporaries                              | caller              |

**Critical divergence from arm64: NO callee-saved FP registers under SysV
AMD64.** All 16 `%xmm` registers are caller-saved. This is the single most
important architectural difference between the two profiles. On arm64, `s8`
and `s9` (lower 32 bits of `v8`/`v9`) are callee-saved FP registers: the
softmax emitter can hold `row_max` in `s8` and `row_sum` in `s9` across the
`bl _expf` call without any memory traffic. On x86_64, there is no equivalent
register — `row_max` and `row_sum` must be spilled to stack before each
`call expf@PLT` and reloaded after. This drives the §4.5 xmm-spill strategy
described below.

## 4. Supported Ops

Full op-parity with arm64 minus SIMD/AVX:

| StdOp                      | Supported | Notes                                                       |
|----------------------------|-----------|-------------------------------------------------------------|
| `Linear` (no `bias` attr)  | yes       | Scalar matmul. Fuses `PostOp::Relu` (inline `maxss`) and `PostOp::SoftmaxRow` (separate i-loop B). |
| `Linear` (`bias=true`)     | yes       | Matmul + per-output bias-add inline. Fuses same post-ops. |
| `Relu`                     | yes       | Standalone elementwise loop, copy-with-clamp `src→dst`. Default: fused into preceding Linear. |
| `Dropout`                  | yes       | No asm; `BufferLoc::Alias(operand)` propagation. Default: removed by `EliminateDropout`. |
| `Softmax`                  | yes       | 3-pass per-row (max → exp+sum → normalise), `call expf@PLT`. Default: fused as `PostOp::SoftmaxRow`. |
| `Input`                    | yes       | Marker only — `BufferLoc::InputReg` (`%rdi`).              |

**Absent from this profile (see §8):** AVX/AVX-512, FMA, vectorised exp,
macOS Mach-O symbol form, Windows x64 ABI.

## 5. Fused Softmax xmm-Spill Strategy

### Architectural rationale

Under SysV AMD64, `call expf@PLT` may clobber all 16 `%xmm` registers
(they are caller-saved). The softmax emitter needs `row_max` and `row_sum`
to survive the call. Because there are no callee-saved FP registers, the
only option is to spill to stack.

### Stack slot layout

`assign_buffers` owns two fixed-offset stack slots at the bottom of every
model frame that calls extern math (`model.calls_extern_math()` returns `true`).
The slots are pinned at:

| Slot         | Address       | Offset | Purpose              |
|--------------|---------------|--------|----------------------|
| `row_max`    | `(%rsp)`      | 0      | Per-row maximum      |
| `row_sum`    | `8(%rsp)`     | 8      | Per-row sum          |

The 16-byte reserve is initialised by `assign_buffers`: when
`model.calls_extern_math()`, `stack_offset` starts at `16` (not `0`),
shifting all `BufferLoc::StackOffset` values up by 16. This anchors the
spill addresses across all models — including those with non-empty intermediate
buffers — without per-emitter parameterisation, and prevents the slot/buffer
overlap that would corrupt Phase 2 reads if the slots were placed at the
bottom of the intermediate-buffer region.

### Cost vs arm64

The xmm-spill adds **+1 memory traffic per element in Phase 3** (one
`movss (%rsp), %xmmN; movss %xmmN, 8(%rsp)` pair per `call expf@PLT`
iteration) vs arm64's register-resident pattern (no memory traffic across
`bl _expf`). This cost is accepted because:

1. It is the minimal correct solution under SysV AMD64 calling conventions.
2. Any vectorised `expf` approximation (a future SIMD profile) eliminates
   the `call` entirely, making the spill moot.
3. Profile isolation: accepting scalar-with-spill in this profile keeps the
   arm64 profile unchanged.

## 6. Libm Call Form

Softmax emits `call expf@PLT` to invoke the C library's single-precision
exponential function.

**Why `@PLT`?** On Linux ELF, position-independent executables (PIE) and
shared objects cannot call external symbols with a direct 32-bit relative
displacement — the symbol may be more than ±2 GB away, and more critically,
its address is only known at dynamic link time, not at assembly/link time.
The procedure-linkage table (PLT) solves this: the first call goes through a
PLT stub that invokes the dynamic linker to resolve the symbol and patch the
global offset table (GOT); subsequent calls are direct. `@PLT` is the
relocation modifier that makes this mechanism explicit in AT&T syntax.

```asm
call    expf@PLT        # resolve via PLT; first call patches GOT
```

Compare with arm64 Mach-O, where the call is `bl _expf` with no explicit
relocation modifier. On Mach-O, the underscore-prefixed symbol form and the
default flat namespace make the relocation implicit.

**Register usage across the call.** Per SysV AMD64: `%xmm0` carries the
float argument (the `fsub`-adjusted element value) and the return value.
After the call, `%xmm0` holds `expf(x - row_max)`. All other `%xmm`
registers must be assumed clobbered — this is what forces the xmm-spill
described in §5.

## 7. Stack Alignment and `compute_frame_size`

### The 16-byte invariant

SysV AMD64 requires `%rsp` to be 16-byte aligned at the point of each `call`
instruction. At function entry, after the caller's `call` pushed the 8-byte
return address, `%rsp ≡ 8 (mod 16)`.

The prologue pushes N callee-saved registers (8 bytes each). After N pushes:
`%rsp ≡ 8 - 8·N ≡ 8·(1 - N) (mod 16)`.

To restore 0 (mod 16) alignment before any internal `call expf@PLT`, the
frame must be padded:

- If `num_pushes` is **even**: `%rsp ≡ 8 (mod 16)` after pushes → subtract
  8 extra bytes (the `+8` correction).
- If `num_pushes` is **odd**: `%rsp ≡ 0 (mod 16)` after pushes → no
  correction needed.

### `compute_frame_size` formula

```
frame_size = round_up_16(raw_bytes) + if num_pushes % 2 == 0 { 8 } else { 0 }
```

Where `raw_bytes` is the raw intermediate-buffer requirement (from
`assign_buffers`) and `round_up_16` rounds up to the next 16-byte multiple.

The formula is derived in spec §7.5 with 8 annotated unit-test cases covering
zero/nonzero raw bytes and both even/odd push counts. `compute_frame_size`
has its own 8-test suite in `profiles/x86_64/src/tests.rs`.

### Standard prologue

For a model that uses softmax (non-leaf, all 6 callee-saved regs pushed):

```asm
    pushq   %rbp
    pushq   %rbx
    pushq   %r12
    pushq   %r13
    pushq   %r14
    pushq   %r15
    subq    $<frame_size>, %rsp
```

Epilogue is the symmetric reverse (`addq` then 6 `popq` in reverse order,
then `ret`).

For leaf models (no `call expf@PLT`): only `pushq %rbp` / `popq %rbp` plus
`subq`/`addq` for intermediate buffers if any. Leaf detection is via
`compute_is_leaf` in `buffer.rs`.

## 8. Out of Scope

The following are explicitly not implemented in this profile:

- **SIMD / AVX / AVX-512 / FMA** — Scalar throughout. Vectorised codegen
  is a future profile (Axis 1 follow-up, post-M9).
- **macOS x86_64 (Mach-O)** — Different symbol prefix (`_`), different
  linker defaults, different `@PLT` handling. A future `x86_64-macos`
  profile would reuse this profile's codegen with `sym_prefix() -> "_"`
  and `expf` vs `expf@PLT`.
- **Windows x64 ABI** — Different register convention (first 4 args in
  `%rcx/%rdx/%r8/%r9`), shadow space requirement, different callee-saved
  set. Out of scope for v1 profiles.
- **Bare-metal `expf`** — This profile calls `expf@PLT` (libm) and requires
  a dynamic linker at runtime. A Taylor-series or minimax `expf` for
  bare-metal targets is Axis 3 work (spec §12 OQ-3), independent of
  which profile implements it.

## 9. Adding a New Op

To add an op to the `x86_64` profile (e.g. `tanh`, `sigmoid`):

1. Add an arm in `profiles/x86_64/src/codegen.rs::classify_op`.
2. Add a per-op emitter as `profiles/x86_64/src/ops/<op>.rs` exposing
   `pub fn emit_<op>(...)`.
3. Re-export the emitter from `profiles/x86_64/src/ops/mod.rs`.
4. Add a dispatch arm in `walk_model`'s op-loop.
5. If the op emits a `call`, update `compute_is_leaf` / `compute_callee_saved`
   in `buffer.rs`, and update `assign_buffers` if the op needs stack slots
   (following the xmm-spill ownership model in §5).
6. Add unit tests in `profiles/x86_64/src/tests.rs`.
7. Add an integration test in `profiles/x86_64/tests/integration.rs` (guard
   with `#![cfg(all(target_os = "linux", target_arch = "x86_64"))]`).
8. Update §4 of this doc.

## 10. M10 ops

M10 adds two new ops (`StdOp::Matmul`, `StdOp::MulScalar`) and generalises
softmax dispatch from rank-2 to rank ≥ 2. The new ops live in their own
`ops/` modules; the existing `emit_linear` is **unchanged** (§6.1
architectural invariant — adding multi-dim matmul does NOT alter the
hot path for `linear`-as-2D).

### `emit_matmul` — multi-dim matmul over rank ≥ 2 inputs

Source: `profiles/x86_64/src/ops/matmul.rs`. Emitted for every
`StdOp::Matmul` node — typically two per attention block (Q · Kᵀ for
scores, attn · V for the attended values).

**Outer-loop wrapper.** Symmetric to arm64 §M10/`emit_matmul`:
`leading_count = product(shape[..rank-2])`; for 2D inputs the outer
loop runs once and is effectively elided. Inside each outer iteration,
the kernel is the standard scalar AT&T x86_64 matmul.

**`mulss + addss` — no FMA, intentional divergence from arm64.**
The arm64 inner kernel uses `fmadd s0, s1, s2, s0` (single-rounding
FMA). The x86_64 inner kernel uses two scalar instructions:

```asm
    movss   (%r8, %rax, 4), %xmm1     # a[i,k]
    movss   (%r9, %rcx, 4), %xmm2     # b[k,j]
    mulss   %xmm1, %xmm2              # tmp = a*b (separate rounding)
    addss   %xmm2, %xmm0              # sum += tmp (separate rounding)
```

This is **deliberate** — `emit_linear` for x86_64 made the same call
in M9 to keep SSE2-only as the floor (FMA requires the FMA3 ISA
extension, post-Sandy-Bridge). Note that this means cross-profile
bit-exact at the byte level is unreachable for any model containing
matmul: the FMA divergence introduces a single ULP of difference per
inner-loop step (§7.2 acceptance criterion: per-profile bit-exact
only).

**Register layout.** `%r8` (A base), `%r9` (B base), `%r11` (DST
base) are materialised once before the outer loop and MUST NOT be
mutated inside it. Per-outer-iteration slice pointers reuse `%rdi`
(A_slice), `%rsi` (B_slice), `%rdx` (DST_slice) — i.e., the FFI
argument registers. To preserve the FFI calling convention for
downstream emitters (e.g. a subsequent `emit_matmul` re-materialising
from `InputReg` / `OutputReg`), `emit_matmul` defensively spills
`%rdi`/`%rsi`/`%rdx` into `%xmm8`/`%xmm6`/`%xmm7` at function-body
entry and restores at exit:

```asm
    movq    %rdi, %xmm8           # preserve %rdi (input ptr)
    movq    %rsi, %xmm6           # preserve %rsi (params ptr)
    movq    %rdx, %xmm7           # preserve %rdx (output ptr)
    ; ... outer loop body ...
    movq    %xmm8, %rdi           # restore
    movq    %xmm6, %rsi
    movq    %xmm7, %rdx
```

The xmm-spill triple is the canonical M9 idiom (commits `ecb69ac`,
`c3ff521`) — cheap (3 movq instructions at entry, 3 at exit) and
free of stack-frame impact. The `%rdi` spill is the M10 fixup
(commit `aac8650`): the attention fixture has two matmuls that both
materialise B from `x` (InputReg → `%rdi`); without the spill the
first matmul's outer loop overwrites `%rdi` with an A_slice pointer,
silently miscompiling the second matmul.

### `emit_mulscalar` — flat per-element scalar multiply

Source: `profiles/x86_64/src/ops/mulscalar.rs`. Emitted for every
`StdOp::MulScalar` node — one per attention block (the `1/√d` scaling
of the `Q · Kᵀ` scores before softmax).

**Pre-loaded scalar via `movl + movd`.** The scalar (an f32 bit
pattern computed by the dispatcher from the AttrValue) is materialised
into `%xmm4` once before the loop:

```asm
    movl    $0x<scalar_bits>, %r10d
    movd    %r10d, %xmm4
```

**Inner `mulss` loop.** With `BufferLoc::Alias` plumbing
`src_loc == dst_loc`, the loop reads and writes the same buffer:

```asm
    movss   (%r8, %rcx, 4), %xmm0
    mulss   %xmm4, %xmm0
    movss   %xmm0, (%r11, %rcx, 4)        # %r11 == %r8 under Alias
```

**`AttrValue::Float → f32` truncation contract.** Same as arm64: the
dispatcher (`codegen.rs`) truncates `f64 → f32`, transmutes to `u32`
bits via `f32::to_bits`, and passes the `u32` to `emit_mulscalar`.
Per-profile bit-exact is guaranteed because both profiles consume the
same `u32`.

### Softmax dispatch — generalised to rank ≥ 2

`emit_softmax` itself is structurally unchanged at the body level (the
3-pass max → exp → normalise is per-row, regardless of rank), but the
dispatcher in `codegen.rs::walk_model` now computes:

- `b = product(shape[..rank-1])` — total number of rows across all leading dims.
- `k = shape[rank-1]` — row width (last dim).

For 2D inputs this collapses to the M3/M9 shape (`b = batch`,
`k = features`). For the 4D attention case the leading dims (batch,
heads, seq) flatten into a single outer counter — softmax is applied
to the **last axis only**.

**Post-Group-10 codegen fix.** During M10 end-to-end FFI integration
(commit `feb65de`, Group 10) we discovered that `emit_softmax` itself
was clobbering `%rdi`/`%rsi`/`%rdx` (the FFI input/params/output
registers) because `call expf@PLT` may clobber all caller-saved
registers (which on SysV AMD64 includes all the integer arg
registers). Any downstream emitter that re-materialises from
`InputReg`/`OutputReg` after softmax (e.g. attention's second
`emit_matmul` reading from the same input `x` as the first) was
silently miscompiling. Fix: `emit_softmax` now `pushq %rdi`,
`pushq %rsi`, `pushq %rdx` plus a `pushq %rax` padding push at
function-body entry, and matches with `popq %rax`, `popq %rdx`,
`popq %rsi`, `popq %rdi` at exit.

The padding push has a side effect: the row_max / row_sum stack slots
(§5) shifted from `(%rsp)` / `8(%rsp)` to `32(%rsp)` / `40(%rsp)` to
account for the 4 extra 8-byte pushes (`4 * 8 = 32` bytes added to
the offset). The slot ownership in `assign_buffers` is unchanged in
shape (still a 16-byte reserve at the bottom of the frame), but the
offsets used by `emit_softmax` are shifted by `+32`. This is the M9
xmm-spill lesson re-applied.

### `f32::exp` divergence: cross-profile non-goal

Even with both profiles producing per-profile bit-exact output, the
two profiles disagree at the byte level on any model containing
softmax. glibc's `expf` (Linux x86_64) and Apple libsystem's `expf`
(macOS arm64) implement `f32::exp` to slightly different precisions —
typically diverging in the last 1-2 ULP. The M10 acceptance criterion
(§7.2) is **per-profile bit-exact** only; cross-profile bit-exact is
deferred to a future tolerance-report harness (OQ-BENCH+).

### What is unchanged

- `emit_linear` body is byte-identical to M9. `linear`-as-2D models
  (every M3-M9 fixture) lower exactly as before.
- `emit_relu`, the prologue/epilogue analyzers, the buffer-allocation
  strategy, and the `compute_callee_saved` predicate — all unchanged.
  M10 added two new ops, one dispatcher generalisation, and a
  softmax FFI-register preservation block; the surrounding scaffolding
  is the same.

## 11. Multi-Input ABI (M12)

M12 adds per-profile `AbiContext` that extends the calling convention to N∈{1..4}
inputs (where the hardware scratch budget allows — see §11.5 for the
N=4 + matmul history on this profile, closed in M13).

### 11.1 Register layout (SysV AMD64)

Under SysV AMD64, the first six integer/pointer arguments are passed in
`%rdi/%rsi/%rdx/%rcx/%r8/%r9`. The M12 convention assigns registers as follows
for N inputs:

| N | %rdi | %rsi | %rdx | %rcx | %r8    | %r9 |
|---|------|------|------|------|--------|-----|
| 1 | in₀  | params | out | —  | —      | —   |
| 2 | in₀  | in₁  | params | out | —    | —   |
| 3 | in₀  | in₁  | in₂  | params | out | —  |
| 4 | in₀  | in₁  | in₂  | in₃  | params | out |

`params` always occupies register `[N]` (0-indexed) and `out` occupies `[N+1]`.
N=1 is the legacy single-input layout and is fully backward-compatible with all
pre-M12 fixtures.

### 11.2 `AbiContext` — per-arity accessor struct

`profiles/x86_64/src/abi.rs` exports `AbiContext { n_inputs: usize }` with the
following arity-aware accessors:

- `input_reg(i: usize) -> &'static str` — returns `"%rdi"` through `"%rcx"` for
  `i` in `0..n_inputs`. Panics if `i >= n_inputs`.
- `params_reg() -> &'static str` — returns `"%rsi"` through `"%r8"` (= `arg[n_inputs]`).
- `output_reg() -> &'static str` — returns `"%rdx"` through `"%r9"` (= `arg[n_inputs+1]`).
- `ffi_save_set() -> &[&'static str]` — returns the integer registers that must
  be preserved across the outer matmul loop body (all ABI arg registers that
  would otherwise be clobbered when reused as per-iteration slice pointers).
- `materialise_ptr(b: &mut String, dst: &'static str, loc: &BufferLoc, ...)` —
  emits the `movq`-based materialise sequence appropriate for the given
  `BufferLoc` variant.

`walk_model` constructs exactly one `AbiContext { n_inputs: model.inputs.len() }`
and threads `&abi` through every op-emitter call. Models with more than 4 inputs
return `Err(LowerError::TooManyInputs { n: model.inputs.len() })`.

### 11.3 Stack alignment with `pushq %rax` padding

When `ffi_save_set()` has an **odd** cardinality, `emit_ffi_save` prepends an
extra `pushq %rax` to keep `%rsp` 16-byte aligned before any `call expf@PLT`.
`emit_ffi_restore` mirrors this with a matching `popq %rax` at the start of the
restore sequence.

`emit_ffi_restore` reverses `emit_ffi_save` in strict **LIFO** order: if the
save sequence was `pushq %rdi; pushq %rsi; pushq %rdx`, the restore sequence is
`popq %rdx; popq %rsi; popq %rdi`.

### 11.4 `emit_matmul` rework — option β (callee-saved scratch)

In M10, `emit_matmul` spilled `%rdi/%rsi/%rdx` (the FFI ABI registers at N=1)
into `%xmm8/%xmm6/%xmm7` via `movq` at function-body entry. This was safe under
N=1 because `%r8`/`%r9` (unused by the N=1 ABI) were available as base-pointer
scratch.

Under M12's expanded ABI window, `%r8` and `%r9` may carry `params` or `output`
at N=3/N=4, leaving only three caller-saved non-ABI scratch registers for a
3-register base-pointer set. The xmm-spill approach is still used for the ABI
save/restore block (option α — cheap, 3 movq each way), but per-outer-iteration
slice pointers are moved to **callee-saved** registers (`%rbx`, `%r12`, `%r13`,
`%r14`) whose save/restore is handled by the existing prologue/epilogue
controlled by `compute_callee_saved`:

| Register       | Role in `emit_matmul` (M12) |
|----------------|------------------------------|
| `%r8`  (or `%rax`) | A-base pointer (materialised once before outer loop) |
| `%r9`  (or `%r11`) | B-base pointer (materialised once before outer loop) |
| `%r11`         | DST-base pointer (materialised once before outer loop) |
| `%rbx`         | A_slice pointer (per outer-iteration, callee-saved) |
| `%r12`         | B_slice pointer (per outer-iteration, callee-saved) |
| `%r13`         | DST_slice pointer (per outer-iteration, callee-saved) |
| `%r14`         | outer loop index (callee-saved) |

`compute_callee_saved` is extended in M12 to include `%rbx/%r12-%r14` whenever
`emit_matmul` is present in the model, triggering their save/restore in the
function prologue/epilogue.

The old `movq %rdi, %xmm8 / movq %rsi, %xmm6 / movq %rdx, %xmm7` outer-loop
spill block from M10 is fully eliminated. The materialise-ptr-first ordering
rule applied at `walk_model` call-site level ensures base pointers are loaded
before any slice arithmetic clobbers scratch.

### 11.5 N=4 + matmul gap — closed in M13

**N=4 + matmul is fully supported on x86_64 as of M13.** The M12 limitation
(j-counter `%r9` colliding with `output_reg()` at N=4) was resolved by
relocating the j-counter to `%rbp` — see §M13 below for details. The
`Err(LowerError::UnsupportedOp)` reject path is removed; N=4 + matmul now
compiles and runs bit-exact (verified by `tests/fixtures/four_input_matmul.nfl`).

## M13 ops

### `emit_add` (`profiles/x86_64/src/ops/add.rs`)

Flat elementwise tensor addition: `dst[i] = a[i] + other[i]` over
`total_elements = product(shape)`.

**Register layout:**
- `%rax` — `a` pointer (caller-saved non-ABI for all N ∈ [1,4],
  materialised via `AbiContext::materialise_ptr`).
- `%r10` — `other` pointer (same).
- `%r11` — `dst` pointer (same).
- `%rbp` — loop counter. **Callee-saved by the function-level
  prologue's unconditional `pushq %rbp`** (asm.rs:66); the body is
  free to clobber it as scratch since no op-emitter reads `%rbp`.
- `%xmm0`, `%xmm1` — load-load-addss-store scalar FP registers.

**Why `%rbp` and not a regular GPR scratch?** The intersection of
free non-ABI scratch GPRs across N ∈ [1,4] is `{%rax, %r10, %r11}` —
exactly the materialised pointer set, leaving zero spare GPR for a
counter. `%rbp` is the simplest exception: it's already saved by the
prologue, restored by the epilogue, and read by zero op bodies.

Inner loop (per iter):
```
movss   (%rax, %rbp, 4), %xmm0   # load a[i]
movss   (%r10, %rbp, 4), %xmm1   # load other[i]
addss   %xmm1, %xmm0
movss   %xmm0, (%r11, %rbp, 4)   # store dst[i]
addq    $1, %rbp
```

### M13 N=4 + matmul fix (note in `emit_matmul`)

M12 (commit 37868e5) capped `emit_matmul` at N≤3 because the inner
j-loop counter was hardcoded to `%r9`, which becomes `output_reg()`
at N=4 (`INPUT_REGS[5]`). M13 (Task 1) relocates the j-counter to
`%rbp` for the same reason `emit_add` uses it: callee-saved by the
prologue, unread by op bodies. The reject path is removed; N=4 +
matmul now compiles and runs bit-exact (verified by
`tests/fixtures/four_input_matmul.nfl`).
